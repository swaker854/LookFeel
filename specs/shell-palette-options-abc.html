Shell Palette — Options A / B / C

Three structural arrangements of the same hues. Semantic demo shows how status badges read in each option.

Option A Status Quo — 3 support + 4 semantic 8 accent hues

Support family (violet / rose / teal) used only for structural/categorical meaning in areas like Data Model. Semantic family (green / amber / red / blue) reserved for states. Clean separation of roles.

Maximum role clarity — support never means a state Conventional semantic colors — green/amber/red/blue are universal 8 accent hues is a lot — risk of visual noise Teal and blue read as the same "cool" family in dense UI

Option B Teal collapses into Info — blue retired 6 accent hues

Teal replaces blue as the info semantic. The "fourth" structural slot is eliminated. Support family shrinks to violet + rose. Fewer total hues, and teal reads naturally as informational.

6 hues instead of 8 — meaningfully leaner Teal is a natural informational color in many systems Eliminates teal/blue near-collision from Option A Breaks blue=info convention — may feel unfamiliar Teal doing double duty (structural + semantic) blurs the role boundary Lose a structural slot — only 2 categorical colors for Data Model etc.

Option C Violet promoted to 5th semantic (In Progress) 7 accent hues

Violet moves from structural to semantic, gaining the "in-progress" or "neutral-alert" slot used in tools like GitHub, Linear, and Jira. Support family shrinks to rose + teal only. Info stays blue. Adds a useful state that the current 4-semantic system lacks.

Violet/purple is the universal "in-progress" color in BI/PM tools Adds a genuinely useful 5th semantic state — often requested Info stays blue — zero convention break 7 accents — still fairly broad Support shrinks to 2 — rose + teal only for categorical use A 5th semantic state needs an agreed name and clear usage rules
